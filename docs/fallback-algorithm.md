# Резервный алгоритм

## Назначение

`Fallback Recommendation Module` формирует рекомендации, если основной ML-источник недоступен, работает слишком медленно или вернул результат недостаточного качества.

Алгоритм не использует заранее закэшированные ML-предсказания. Он использует только внутренние данные, которые доступны в момент запроса:

- интересы пользователя из профиля;
- историю просмотров, бронирований, оценок и отклонений;
- текущий город или направление поездки;
- даты поездки;
- тип рекомендаций;
- агрегированную популярность объектов среди анонимизированных пользователей.

## Входные данные

| Данные | Источник |
|---|---|
| `userId` | API-запрос |
| `tripId` | API-запрос |
| `recommendationType` | API-запрос |
| `limit` | API-запрос |
| Интересы и настройки | `User Profile Service` |
| История действий | `User History Service` |
| Популярность объектов | `Popularity Service` |
| Направление и даты поездки | данные поездки, доступные `Recommendation Service` |

## Шаги алгоритма

1. Получить параметры поездки: направление, даты, состав путешествия, если доступен.
2. Получить список доступных объектов для выбранного направления и `recommendationType`.
3. Исключить недоступные объекты.
4. Исключить уже забронированные или явно отклоненные пользователем варианты.
5. Для каждого объекта рассчитать признаки:
   - `profileMatch` - совпадение с интересами пользователя;
   - `historyMatch` - соответствие прошлым просмотрам, бронированиям и оценкам;
   - `tripContextMatch` - соответствие направлению, датам и типу поездки;
   - `popularity` - агрегированная популярность объекта.
6. Рассчитать итоговый балл.
7. Отсортировать объекты по итоговому баллу по убыванию.
8. Вернуть первые `N` вариантов, где `N = limit`.

## Формула

```text
fallbackScore =
    profileMatch * 0.35
  + historyMatch * 0.25
  + tripContextMatch * 0.25
  + popularity * 0.15
```

Веса являются настраиваемыми и должны быть подтверждены владельцем продукта или аналитиком данных.

Каждая часть формулы нормализуется в диапазоне от `0` до `1`. Итоговый `score` также приводится к диапазону от `0` до `1`.

## Правила для недостатка данных

| Ситуация | Правило |
|---|---|
| Нет истории | Использовать профиль, параметры поездки и популярность. |
| Нет профиля | Использовать историю, параметры поездки и популярность. |
| Новый пользователь без истории | Использовать профиль, параметры поездки и популярность. |
| Нет профиля и истории | Использовать параметры поездки и популярность. |
| Нет популярности, но есть профиль или история | Вернуть результат по доступным данным, если найдено достаточно объектов. |
| Нет всех внутренних источников | Вернуть контролируемую ошибку `503`. |

## Reason codes

| Код | Когда используется |
|---|---|
| `MATCHES_USER_INTERESTS` | Объект совпадает с интересами из профиля. |
| `SIMILAR_TO_USER_HISTORY` | Объект похож на ранее просмотренные или забронированные. |
| `MATCHES_TRIP_CONTEXT` | Объект подходит по направлению, датам или параметрам поездки. |
| `POPULAR_IN_DESTINATION` | Объект популярен в выбранном направлении. |

## Независимость от ML

Алгоритм не зависит от `ML Recommendation Service` и не использует готовые ML-предсказания. Он строит результат на внутренних сервисах и простых правилах ранжирования. Даже если персональных данных мало, система может вернуть популярные объекты по направлению и типу рекомендации.
