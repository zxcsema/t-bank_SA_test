# Архитектурное решение

## Компоненты и ответственность

| Компонент | Ответственность |
|---|---|
| Client Application | Отправляет запрос на рекомендации и отображает результат пользователю. |
| Recommendation Service | Принимает API-запрос, проверяет параметры и права, вызывает ML, принимает решение о fallback, возвращает единый ответ, пишет логи и метрики. |
| ML Recommendation Service | Формирует персонализированные ML-рекомендации и возвращает `confidence`. |
| Fallback Recommendation Module | Формирует rule-based рекомендации внутри `Recommendation Service` без использования готовых ML-предсказаний. |
| User Profile Service | Предоставляет интересы и настройки пользователя. |
| User History Service | Предоставляет историю просмотров, бронирований, оценок и отклонений. |
| Popularity Service | Предоставляет агрегированную популярность объектов по направлению и типу рекомендации среди анонимизированных пользователей. |

## Обработка запроса в Recommendation Service

1. Получить запрос пользователя.
2. Проверить корректность запроса.
3. Проверить доступ пользователя к `tripId`.
4. Отправить запрос в `ML Recommendation Service`.
5. Ожидать ответ ML не более 3 секунд.
6. Проверить HTTP-статус и корректность ответа.
7. Проверить, что список рекомендаций не пустой.
8. Проверить `confidence`.
9. При успешном результате вернуть ML-рекомендации.
10. При ошибке, тайм-ауте, пустом списке или низком `confidence` вызвать `Fallback Recommendation Module`.
11. Вернуть пользователю рекомендации в едином формате.
12. Записать технический результат в логи и метрики.

## Условия переключения на fallback

| Причина | Код причины |
|---|---|
| ML вернул HTTP 5xx | `ML_HTTP_5XX` |
| ML недоступен по сети | `ML_NETWORK_ERROR` |
| ML отвечает дольше 3 секунд | `ML_TIMEOUT` |
| ML вернул некорректный JSON или нарушил контракт | `ML_INVALID_RESPONSE` |
| `confidence` ниже порога | `LOW_CONFIDENCE` |
| ML вернул пустой список | `ML_EMPTY_RESULT` |
| Circuit Breaker открыт | `CIRCUIT_OPEN` |

## Используемые паттерны

### Timeout

`Recommendation Service` ограничивает ожидание ответа ML 3 секундами. После истечения времени ожидание прекращается и запускается fallback.

### Fallback

При проблеме с ML система использует `Fallback Recommendation Module`, который строит рекомендации по внутренним данным: профиль, история, параметры поездки и популярность.

### Circuit Breaker

Если ML часто отвечает ошибками или тайм-аутами, `Recommendation Service` временно перестает отправлять запросы в ML и сразу использует fallback. Это снижает нагрузку на неисправный ML и уменьшает время ответа пользователю.

Стартовые параметры Circuit Breaker: открывать цепь после 5 неуспешных ML-вызовов подряд и пробовать восстановление через 30 секунд.

### Graceful Degradation

Если недоступны отдельные внутренние источники данных, fallback продолжает работу с оставшимися данными:

- нет истории - использовать профиль и популярность;
- нет профиля - использовать историю и популярность;
- нет профиля и истории - использовать параметры поездки и популярность;
- нет всех внутренних источников - вернуть контролируемую ошибку `503`.

## Retry

Повторный вызов ML в рамках одного пользовательского запроса по умолчанию не выполняется. Причина: фиксированный тайм-аут 3 секунды и необходимость быстро вернуть ответ пользователю.

Один быстрый retry можно рассмотреть только для сетевой ошибки, если оставшийся временной бюджет позволяет завершить весь запрос без нарушения SLA. В базовом решении retry не используется.

## Данные и логирование

Для технического контроля создается запись `RecommendationRequestLog`. В ней хранится только техническая информация о запросе и результате. Готовые ML-предсказания и полные персональные данные не сохраняются.

## Модель данных: RecommendationRequestLog

| Поле | Описание |
|---|---|
| requestId | Идентификатор запроса от клиента. |
| userIdHash | Хэш идентификатора пользователя. |
| tripId | Идентификатор поездки. |
| recommendationType | Тип рекомендаций: `HOTEL`, `ACTIVITY`, `ROUTE`. |
| source | Источник результата: `ML` или `FALLBACK`. |
| fallbackReason | Причина fallback, если применимо. |
| responseTimeMs | Общее время ответа API. |
| resultCount | Количество возвращенных рекомендаций. |
| status | Технический статус обработки: `SUCCESS`, `FAILED`. |
| createdAt | Дата и время записи. |
| traceId | Технический идентификатор трассировки. |

### Персональные данные

`userId`, профиль, история действий и настройки пользователя относятся к персональным или чувствительным данным. В лог пишется `userIdHash`, чтобы связать технические события без хранения открытого идентификатора пользователя.

### Что нельзя писать в лог

- паспортные данные;
- платежные данные;
- полный профиль пользователя;
- полную историю действий;
- состав полного ML-ответа;
- готовые ML-предсказания для повторного использования.

### Срок хранения

Технические записи `RecommendationRequestLog` хранятся 90 дней, если внутренние политики хранения данных не требуют иной срок.
