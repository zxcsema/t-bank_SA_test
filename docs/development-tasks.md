# Декомпозиция задач для разработчиков

## TASK-01. Добавить тайм-аут вызова ML-сервиса

**Компонент:** `Recommendation Service`.

При вызове `ML Recommendation Service` сервис ждет ответ не дольше 3 секунд. Если время истекло, ожидание прекращается, фиксируется `ML_TIMEOUT` и запускается fallback.

**Критерии приемки**

- если ML отвечает менее чем за 3 секунды, его ответ обрабатывается;
- если ML не отвечает за 3 секунды, ожидание прекращается;
- запускается резервный сценарий;
- пользователь получает рекомендации или контролируемую ошибку;
- в лог записывается `ML_TIMEOUT`;
- увеличивается метрика ML-тайм-аутов.

## TASK-02. Добавить проверку confidence

**Компонент:** `Recommendation Service`.

Перед возвратом ML-ответа пользователю нужно сравнить `confidence` с настроенным порогом. При низком значении сервис фиксирует `LOW_CONFIDENCE` и переходит на fallback.

**Критерии приемки**

- ML-ответ с `confidence >= threshold` возвращается с `source = ML`;
- ML-ответ с `confidence < threshold` не возвращается пользователю;
- запускается fallback;
- в лог записывается `LOW_CONFIDENCE`;
- увеличивается метрика низкого `confidence`.

## TASK-03. Реализовать резервный rule-based алгоритм

**Компонент:** `Fallback Recommendation Module`.

Модуль формирует рекомендации по внутренним данным: фильтрует кандидатов, рассчитывает `fallbackScore`, сортирует результат и ограничивает список значением `limit`. Если после фильтрации нет подходящих объектов, модуль возвращает ошибку fallback в `Recommendation Service`.

**Критерии приемки**

- уже забронированные и отклоненные объекты исключаются;
- объекты сортируются по `fallbackScore` по убыванию;
- возвращается не больше `limit` объектов;
- каждый объект содержит `entityId`, `entityType`, `title`, `score`, `reasonCodes`;
- готовые ML-предсказания не используются.

## TASK-04. Добавить получение профиля и истории пользователя

**Компонент:** `Recommendation Service`, `Fallback Recommendation Module`.

Fallback получает данные из `User Profile Service` и `User History Service`. При недоступности одного источника сценарий продолжается с частичной деградацией.

**Критерии приемки**

- при доступном профиле он влияет на `profileMatch`;
- при доступной истории она влияет на `historyMatch`;
- при недоступной истории используются профиль и популярность;
- при недоступном профиле используются история и популярность;
- ошибки внутренних источников логируются без раскрытия пользователю.

## TASK-05. Добавить использование агрегированной популярности

**Компонент:** `Fallback Recommendation Module`, `Popularity Service`.

Fallback запрашивает популярные объекты для направления, дат и `recommendationType`. Если `Popularity Service` недоступен, расчет продолжается по профилю и истории, если эти данные есть.

**Критерии приемки**

- популярность влияет на итоговый `fallbackScore`;
- популярность берется из агрегированных обезличенных данных;
- готовые ML-предсказания не используются;
- при недоступном `Popularity Service` fallback продолжает работу, если есть другие данные.

## TASK-06. Унифицировать формат ответа ML и fallback

**Компонент:** `Recommendation Service`.

Клиент получает одинаковую API-модель независимо от источника рекомендаций. ML-ответ и fallback-ответ приводятся к общему контракту.

Если ML-результат нельзя привести к контракту, он считается некорректным и запускается fallback.

**Критерии приемки**

- ответ содержит `requestId`, `items`, `source`, `generatedAt`;
- элементы содержат `entityId`, `entityType`, `title`, `score`, `reasonCodes`;
- для ML указан `source = ML`;
- для fallback указан `source = FALLBACK`.

## TASK-07. Добавить обработку ошибок API

**Компонент:** `Recommendation Service`.

Ошибки API возвращаются в формате `code`, `message`, `traceId`. Технические причины отказа ML не раскрываются пользователю.

**Критерии приемки**

- некорректный запрос возвращает `400 INVALID_REQUEST`;
- отсутствие авторизации возвращает `401 UNAUTHORIZED`;
- отсутствие доступа возвращает `403 FORBIDDEN`;
- отсутствующий пользователь или поездка возвращает `404 NOT_FOUND`;
- невозможность сформировать рекомендации возвращает `503 RECOMMENDATIONS_UNAVAILABLE`.

## TASK-08. Добавить логирование и метрики

**Компонент:** `Recommendation Service`.

Сервис записывает `RecommendationRequestLog` и обновляет метрики по итогам обработки запроса. Ошибка логирования не должна ломать пользовательский сценарий.

**Критерии приемки**

- лог содержит `requestId`, `traceId`, `recommendationType`, `source`, `fallbackReason`, время ответа и количество результатов;
- персональные данные не пишутся в открытом виде;
- метрики увеличиваются для запросов, тайм-аутов ML, ошибок ML, fallback и ошибок fallback.

## TASK-09. Добавить Circuit Breaker

**Компонент:** `Recommendation Service`.

При серии ошибок ML сервис временно открывает цепь и сразу использует fallback. Для таких запросов фиксируется причина `CIRCUIT_OPEN`.

**Критерии приемки**

- после заданного числа ошибок ML новые запросы временно не вызывают ML;
- при открытой цепи запускается fallback;
- причина `CIRCUIT_OPEN` записывается в лог;
- после периода восстановления выполняется пробный вызов ML.

## TASK-10. Добавить тесты резервного сценария

**Компонент:** `Recommendation Service`.

Нужно покрыть unit- и интеграционными тестами успешный ML-ответ, причины перехода на fallback и контролируемые ошибки. В ответах пользователю не должно быть технических деталей ML.

**Критерии приемки**

- тестируется успешный ML-ответ;
- тестируются HTTP 500, сетевая ошибка, тайм-аут, низкий `confidence`, пустой ML-ответ;
- тестируется отсутствие профиля, истории и популярности;
- тестируется `503`, если рекомендации сформировать невозможно.
