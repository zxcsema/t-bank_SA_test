# Анализ требований

## Цель изменения

Обеспечить непрерывность пользовательского сценария подбора рекомендаций в TravelTech, если `ML Recommendation Service` недоступен, отвечает дольше 3 секунд или возвращает результат с низким `confidence`.

## Основной пользователь

Пользователь TravelTech, который планирует поездку и запрашивает рекомендации отелей, активностей или маршрутов.

## Требования из задания

### Функциональные требования

| ID | Требование |
|---|---|
| FR-01 | Система должна принимать запрос на рекомендации по типам `HOTEL`, `ACTIVITY`, `ROUTE`. |
| FR-02 | `Recommendation Service` должен вызывать `ML Recommendation Service` как основной источник рекомендаций. |
| FR-03 | Если ML недоступен, система должна перейти на резервный сценарий. |
| FR-04 | Если ML отвечает дольше 3 секунд, система должна перейти на резервный сценарий. |
| FR-05 | Если ML возвращает низкий `confidence`, система должна перейти на резервный сценарий. |
| FR-06 | Резервный сценарий должен использовать профиль пользователя, историю действий и популярность объектов среди анонимизированных пользователей. |
| FR-07 | Резервный сценарий не должен использовать заранее закэшированные ML-предсказания. |
| FR-08 | Пользователь должен получать рекомендации в едином формате независимо от источника. |
| FR-09 | Пользователю не должны показываться технические детали отказа ML. |

### Нефункциональные требования

| ID | Требование |
|---|---|
| NFR-01 | Тайм-аут ожидания ML должен быть 3 секунды. |
| NFR-02 | Пользовательский сценарий не должен зависать при отказе ML. |
| NFR-03 | API должен сохранять обратную совместимость для клиентов. |
| NFR-04 | Паспортные и платежные данные не должны передаваться в рекомендательные сервисы. |
| NFR-05 | Причины переключения на fallback должны логироваться и попадать в метрики. |

## Рабочие параметры

| ID | Параметр |
|---|---|
| A-01 | Низким считается `confidence < 0.70`, если заказчик не задаст другое значение. |
| A-02 | Целевое время ответа API - не более 5 секунд при нормальной работе внутренних источников. |
| A-03 | Временной бюджет fallback-алгоритма - до 1.5 секунд. |
| A-04 | Поле `limit` принимает значения от 1 до 50, по умолчанию 10. |
| A-05 | `tripId` уже создан в системе планирования поездок и доступен для проверки прав. |
| A-06 | `Popularity Service` возвращает актуальную агрегированную популярность, а не готовые ML-предсказания. |
| A-07 | Веса fallback-формулы задаются конфигурацией. |
| A-08 | До отдельного уточнения один порог `confidence` применяется ко всем типам рекомендаций. |

## Бизнес-правила

| ID | Правило |
|---|---|
| BR-01 | Основной источник рекомендаций - `ML Recommendation Service`. |
| BR-02 | Fallback включается при сетевой ошибке ML, HTTP 5xx, тайм-ауте 3 секунды, некорректном ответе, пустом списке или низком `confidence`. |
| BR-03 | Повторный вызов ML в рамках одного пользовательского запроса не выполняется, чтобы не увеличивать время ожидания. |
| BR-04 | Рекомендации из ML и fallback возвращаются в одном формате. |
| BR-05 | Уже забронированные или явно отклоненные пользователем объекты не должны попадать в fallback-результат. |
| BR-06 | Если история недоступна, fallback использует профиль и популярность. |
| BR-07 | Если профиль недоступен, fallback использует историю и популярность. |
| BR-08 | Если профиль и история недоступны, fallback использует параметры поездки и популярность. |
| BR-09 | Если недоступны все внутренние источники, система возвращает контролируемую ошибку `503`. |

## Ограничения

- Нельзя использовать кэш готовых ML-предсказаний.
- Нельзя передавать паспортные и платежные данные в рекомендательные сервисы.
- Нельзя показывать пользователю техническую причину отказа ML.
- Не добавляются новые компоненты сверх предложенного состава без необходимости.
- Не проектируется отдельная база с результатами рекомендаций.

## Вопросы к заказчику

- Какое значение `confidence` считать низким?
- Одинаков ли порог `confidence` для отелей, активностей и маршрутов?
- Какое максимальное время ответа всей системы допустимо для пользователя?
- Сколько рекомендаций нужно возвращать по умолчанию?
- Можно ли возвращать частичный результат, если найдено меньше `limit` объектов?
- Нужно ли показывать пользователю нейтральную пометку, что рекомендации сформированы по базовым правилам?
- Как долго хранить технические логи запросов рекомендаций?
- Какие действия пользователя считаются явным отклонением объекта?
